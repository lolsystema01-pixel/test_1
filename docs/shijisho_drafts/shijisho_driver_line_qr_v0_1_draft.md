## A. 固定の前提（毎回これを先頭に貼る）

- 新基盤（Supabase ＋ SvelteKit ＋ Cloud Run）。移行ではなく新規構築。
- 検証環境のみ。AI生成のダミーデータを使う。本番データ・現行GASには触らない。
- 全テーブルでRLSを有効にする。秘密情報は環境変数に置く。
- 画面・操作・用語は現行マニュアルv9に合わせる（用語集v0.1）。

## B. この指示書の中身

### タイトル

- ドライバー向けLINE QR返信 v0：ドライバー専用LINE公式アカウントに**12桁の数字**（問合番号のKAZ後ろ）を送ると、ボットが**「KAZ＋12桁」を埋めたQR画像**をトークに返す。ドライバーはスマホ画面のQRをそのままスキャナに読ませる。
- ステートレス（1メッセージ→1返信・会話FSM不要）。**DB非接触・service_role不使用**。QRの中身＝送られた番号そのものなので、追加の情報漏洩がない設計。

### 対象機能（要件定義の章番号）

- **新規の現場補助ツール（要件定義に直接該当章なし）＝本指示書の承認をもって位置づけ確定（LOL）**。
- 関連：§6.7 仕分け（バーコードスキャンの代替）／§8.5 配達前照合（問合番号読取の代替）／第10章 外部IF（LINE）／§11.3 セキュリティ（秘密は環境変数・PIIマスキング）。
- 対象の問合番号体系：先頭 `KAZ` 固定＋数字。**実伝票のバーコード下に印字されている英数字列（例 `KAZ26181000520`）が正の問合番号**であり、QRにはこの値と同じものを埋める。

### 前提（確定事項）

- ドライバー専用LINE公式アカウントは**作成済み**（@608abcuq）。荷受人向け受付ボットとは**別アカウント**。
- 実装は `apps/reception_ui_v0` に**増築**（既存の署名検証・レート制限・PIIマスキングの作法を流用）。ただし**既存の荷受人向け `/webhook/line` には一切手を入れない**（ルート・secret・環境変数を完全分離）。
- 返信方式＝**LINE画像メッセージ**。画像はStorageに置かず、**アプリ自身がオンザフライ生成**（`GET /qr/<番号>.png`）。LINEの画像メッセージは https の公開URLが必要なため、**LINE実機確認にはアプリの公開URL（デプロイ／トンネル）が必要**（【人】）。事前検証はスタブ＋HTTPスモークで行う（受付チャネルv0と同じ作法）。

### やること（具体的に）

凡例：【人】＝手作業／【AI】＝Claude Codeで生成・実行。

- 【人】 LINE Developers でドライバー専用チャネル（@608abcuq）の Messaging API を有効化し、**チャネルシークレット／チャネルアクセストークン**を発行して環境変数に設定する（`DRIVER_LINE_CHANNEL_SECRET` / `DRIVER_LINE_CHANNEL_ACCESS_TOKEN`。値はチャット・リポジトリ・メモに貼らない）。Webhook URL登録・応答メッセージ（自動応答）OFF。友だち追加は**ドライバーに限定する運用**（URL/QRの配布先を絞る）。
- 【AI】 **番号パーサ（純関数）**を作る：入力テキストを正規化（全角数字→半角・前後/途中の空白とハイフン除去・`KAZ`/`kaz` 付き入力も許容して除去）→ **12桁の数字**なら正準値 `KAZ＋12桁` を返す。それ以外は「対象外」を返す（桁数は設定値。既定12桁・要確認欄参照）。
- 【AI】 **QR画像エンドポイント** `GET /qr/[number].png` を作る：`qrcode` npmパッケージで**リクエスト時にPNG生成**して返す（Content-Type: image/png・誤り訂正M・十分な余白）。番号が正準形式（`KAZ`＋数字）でなければ 400。**DBは触らない・無認証**（QRの中身＝URL中の番号そのもの＝新情報を足さないため）。
- 【AI】 **ドライバー用webhook** `POST /webhook/driver-line` を作る：
  - 署名検証＝`DRIVER_LINE_CHANNEL_SECRET`（未設定時は検証スタブ＝通す。受付チャネルv0と同じ）。
  - テキストメッセージをパーサに通し、**12桁なら画像メッセージで返信**（`originalContentUrl`／`previewImageUrl`＝`<PUBLIC_APP_BASE_URL>/qr/KAZ<12桁>.png`）。`DRIVER_LINE_CHANNEL_ACCESS_TOKEN` 未設定時は返信内容をログ出力（スタブ）。
  - 12桁でない入力には**使い方ガイド文**をテキスト返信（例「問合番号の数字12桁を送ってください（KAZは不要）」）。
  - レート制限（既存 `ratelimit.ts` 流用）・ログのPIIマスキング（既存 `mask.ts` 方針）。
- 【AI】 環境変数を `.env.example` に追記：`DRIVER_LINE_CHANNEL_SECRET` / `DRIVER_LINE_CHANNEL_ACCESS_TOKEN` / `PUBLIC_APP_BASE_URL`（QR画像URLの組み立て用）。
- 【AI】 検証を書く：
  - 単体：パーサ（正規化・KAZ付き入力許容・桁数・非数字・対象外）／QR生成（**生成PNGをデコードして中身が `KAZ＋12桁` と一致**することを実証）。
  - HTTPスモーク：webhook（12桁→画像返信ペイロードに `/qr/…png` URL／非12桁→ガイド文／署名不正→403）・`/qr/…png`（正準番号→200 PNG・デコード一致／形式不正→400）。
  - 既存テストの非破壊：受付チャネルv0の全テスト（30/30）が引き続き通ること。
- 【AI/人】 実機確認（承認後・公開URL用意後）：LINEアプリから @608abcuq に12桁を送信→QR画像が返る→**スマホ画面のQRをスキャナ（仕分けナビのカメラ入力等）で読み、`KAZ＋12桁` が取れる**ことを確認。結果を業務Bへ記録。

### やらないこと（範囲外）

- **DB照会・番号の存在チェック・ドライバー本人認証**（LINE userId↔driver_id紐付け）＝温度感未定のため後回し（要確認欄）。v0は形式チェックのみ。
- 荷受人向け受付チャネル（`/webhook/line`・FSM・N-3/N-4）の変更。再配達受付・状況照会との連動。
- QRの印刷・ラベル化（§6.8 ラベル印刷は別系統）。Supabase Storageへの画像保存。
- 仕分けナビ・配達前照合側の改修（QRは既存スキャンで読める前提。読めない場合は要確認欄へ）。
- 本番データ・現行GAS・本番キー。LINEアカウントの本番運用設計（配布・権限管理の作り込み）。

### 合格条件（動作確認の観点）

1. 12桁の数字を送ると**QR画像が返る**（事前検証＝送信ペイロードのURL実証、実機＝トークに画像）。
2. 返ったQRをデコードすると **`KAZ＋12桁` に一致**する（単体テストで実証。実機はスキャナ読取で確認）。
3. 正規化が効く：`kaz900000000001`・空白/ハイフン混入・全角数字でも受理して同じ正準値になる。
4. 12桁でない・数字でない入力には**使い方ガイド文**が返り、QRは返らない。
5. `GET /qr/KAZ<12桁>.png` が 200／image/png。形式不正の番号は 400。
6. 署名不正のwebhook呼び出しは 403。レート制限が効く（上限超過で抑止）。
7. **DB非接触・service_roleキー不使用・ログにPII/シークレット値が出ない**。既存 `/webhook/line`（荷受人向け）が無改修で従来どおり動く（既存テスト全通過）。
8. svelte-check 0エラー・build成功。

### 前提・要確認（LOL・実データ確認待ち）

1. **桁数**：ヒアリングでは「KAZ＋12桁」。ただし実伝票サンプルの送り状番号は `KAZ26181000520`＝**KAZ＋11桁**だった。パーサの桁数は設定値（既定12）とし、実データで確定後に固定する。→【要確認】
2. **現場スキャナがQR（2次元コード）を読めるか**：伝票の1Dバーコード専用ハンディだと読めない可能性。仕分けナビのカメラ入力（BarcodeDetector）はQR可。→【要確認】
3. **認証・存在チェックの温度感**：v0はなし（友だち追加をドライバーに限定する運用のみ）。本人認証（LINE userId↔driver_id）・DB存在チェックを足すかは後続判断。→【要確認・後回し】
4. **本機能の位置づけ（章番号）**：要件定義外の新規補助ツール。本指示書の承認をもって確定。→【LOL承認事項】
5. **公開URL**：LINEが画像を取得できるデプロイ先（Cloud Run等）またはトンネル。実機確認のタイミングで【人】が用意。
