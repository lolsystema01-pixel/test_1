# 住所正規化・共通ID判定（全国配分の土台）v0.2

指示書 `shijisho/shijisho_address_match_v0_2.docx` の成果物。
配送先住所を正規化し、全国Master／ZonePlanと突合して**共通IDを判定・付与**する。
要件定義 6.2（全国配分：住所→共通ID／階層マッチ／dry-run→本実行／未登録・保留）。

## 前提

- 全国Master／ZonePlan 読込済み（`master_zoneplan_v0/`）。
- 荷物（deliveries）取込済み・未配車・common_id 未付与（`csv_import_v0/`）。

## 成果物 / 実行順（SQL Editor）

| # | ファイル | 役割 |
|---|---------|------|
| 1 | `normalize_v0.sql` | 正規化関数＋未登録住所テーブル＋郵便番号補助スタブ |
| 2 | `match_v0.sql`（A→B） | **A: dry-run**（件数のみ）→ **B: 本実行**（共通ID付与・未登録・保留） |
| 3 | `check_match_v0.sql` | 件数・付与・未登録・保留・共通ID→ゾーン/拠点結合の確認 |

## マッチの考え方

- 正規化（`normalize_addr`）を**荷物側とMaster側に同一適用**：空白除去（全角/半角）・全角数字→半角・ハイフン統一。
- 正規化住所が Masterの「**都道府県+自治体+町名**」で**前方一致**するか。
  - 番地・建物・丁目・表記ゆれ（全角スペース等）は前方一致で自然に吸収。
- 階層マッチ：①丁目一致（Masterに丁目がある場合）→ ②**TownKey（自治体+町名）フォールバック**。
  - 今回 **Masterの丁目は全て空**なので、一致は全て②TownKeyフォールバック（＝合格条件「丁目なし/不一致がTownKeyで当たる」を実証）。

## 期待結果（入力16件）

| 区分 | 件数 |
|------|------|
| TownKeyフォールバック一致（共通ID付与） | **14** |
| 不一致（未登録住所＋保留） | **2**（`存在しない町` / `名古屋市栄`） |
| 丁目一致 | 0（Masterに丁目データが無いため。想定どおり） |

- 付与後：14件は `common_id` あり・status 未配車／2件は `common_id` NULL・status **保留**・`unregistered_addresses` に記録。
- 共通ID → `zone_plan` でゾーン番号・拠点・隣接が引ける。

## 割り切り（範囲外）

- 配布（拠点・営業所への帰属付与）、配車、仕分け、帳票は範囲外。
- 未登録住所の修正UI・再判定フロー、マスタ整合性検査（6.3）は範囲外（記録＋保留まで）。
- 郵便番号データは**補助のスタブ**のみ（日本郵便データの完全取込・更新運用は範囲外）。ダミー荷物に郵便番号列が無いため、判定は町名マッチにフォールバック。
- 丁目の漢数字（一丁目 等）の統一は反復対象（今回のダミーには無い）。
