## 【監査の副産物】area_master の文字化け（指示書v1.1のスコープ外・別バグ）

### 事実
| 対象 | 壊れた行 |
|---|---|
| 取込元CSV（`エリアマスタ_最終3_nobom.csv`） | **0**（正当なUTF-8・置換文字なし） |
| `area_master_staging` | **13**（prefecture/municipality/town のいずれか） |
| `area_master`（全81,196行） | `town_key` **13** ／ `source_town_key` **20** ／ pref 3・muni 5・town 5 |

`3+5+5 = 13 = townkey_bad` が一致 → **1行につき壊れているのは1フィールドだけ**
（マルチバイト文字の途中でチャンク境界が割れた痕跡。U+FFFD が1バイトずつ入る）。

### 原因
**CSVは無傷。Supabaseへの取込（CSVインポート）時に壊れた。** SQL側の問題ではない。

### 影響
- `town_key` が壊れた **13行** … ②の住所判定（town_key への前方一致）で**永久にマッチしない**
  → 該当町（朝日町大家庄・豊前市中村・熊本市東区八反田・塩尻市広丘郷原・周南市川端町 等）宛の荷物が
    共通IDを得られず**静かに保留へ落ち続ける**。🔴 要修正
- `source_town_key` のみ壊れた **20行** … 判定に使わない属性列。🟡 表示・追跡上の問題

### 修正方針（検討済み）
- ❌ `source_town_key` からの自己修復は**不可**。CSV原本で `src_town_key` と `prefecture|municipality|town` が
  **856行不一致**（元システム由来の別キー。例: `京都府|京都市中京区|石橋町` vs 実列は上京区）。
  これで復元すると別の町に化ける。
- ✅ **クリーンなCSV原本との突合**で修正する。手順:
  1. `area_master_staging` の壊れた行を CSV の正しい値で UPDATE
     （※ staging に残したまま load を再実行すると復活するため、staging を先に直す）
  2. `area_master` の壊れた行を DELETE（`town_key` が壊れており upsert では消えない）
  3. `area_master_load_v0.sql` の upsert を再実行
  4. 両テーブルで置換文字 0 件を再検証
