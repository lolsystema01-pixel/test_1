### resolve_callback 最終シグネチャ

```sql
resolve_callback(
  p_call_id bigint,
  p_note    text default null,
  p_status  text default '完了'
) returns jsonb
```

- 戻り値（成功時）: `{"result":"resolved","call_id":<id>,"callback_status":<p_status>,"callback_by":<uuid>}`
- 戻り値（対象外・冪等時）: `{"result":"already","call_id":<id>}`
- `p_call_id`・`p_note`のみの2引数呼び出し（既存 `check_call_log_v0.sql` の呼び出し方）は後方互換（`p_status`はデフォルト`'完了'`で動作）。pglite・手動検証（check_call_log_v0.sql相当のブロック実行）の両方で確認済み。
- 旧シグネチャ`resolve_callback(bigint, text)`は`drop function if exists`で明示的に削除してから新シグネチャを作成（オーバーロード共存によるdefault引数解決の事故を防止・冪等）。

### M-1（見送り）の判断

`duplicate`分岐から`call_id`を落とす修正は**見送り**、確認結果メモに「軽微・許容」として記録する。理由：
1. **実効性が薄い**：`record_call_log`は新規`call_sid`ごとに`recorded`分岐で毎回`call_id`（連番）を返す。攻撃者は任意の`call_sid`で新規INSERTを繰り返すだけで連番の増分を観測でき、`duplicate`分岐を経由しなくても総件数の推定は成立する。`duplicate`分岐だけ`call_id`を隠しても、漏洩経路の主開口部（`recorded`分岐）は塞がらない。
2. **契約変更のコスト**：`duplicate`はcall_sidの再送に対する冪等応答であり、会話エンジン（❸・実装未着手）が再送時に同じ`call_id`を受け取れることに依存する設計もあり得る。現時点でこの依存の有無を確認できないため、戻り値の形を変えるのは時期尚早。
3. 根本対策（連番idを外部に一切見せない）をやるなら、`call_id`ではなく`call_sid`のみを返す・またはid列自体を外部非公開にする設計変更が必要で、これは「局所修正」の範囲を超える（テーブル大枠設計に踏み込む）。

→ 結論：今回は見送り。総件数推定のリスクは`recorded`分岐にも同様に存在するため、根治するなら❸実装時に「外部公開する識別子はcall_sidのみとし、call_idは内部専用にする」設計判断をLOL/設計まとめ役に諮ることを推奨（本メモに残置）。

## 備考・設計判断

- **RLSは役割ベース**（`my_role() in ('hq','depot','area')`）。deliveries継承ではない＝通話は着信時点でどの営業所の荷物か不明なため、CS横断で全件可視という業務要件をそのまま反映（指示書の承認事項）。hq/depot/areaは同じ可視件数（絞り込みなし）になる。
- 用件分類 `intent` は列挙値checkなしの自由文字列（カテゴリ確定は運用の細かい仕様のため）。`outcome`/`channel`/`callback_status`のみ列挙値check制約。v0.2で`intent`含む短い分類列に長さCHECKを追加（下記「追加した列長CHECK制約」参照）。
- `record_call_log` は authenticated＋anon 両方から実行可（会話エンジンが未認証セッションで動く想定）。引数に `channel`（default ai_phone）・`started_at`/`ended_at`/`duration_sec` を含む。`resolve_callback` は authenticatedのみ＋関数内で`auth.uid()`必須＋`my_role()`判定の二重防御・`callback_by=auth.uid()`／`callback_note=p_note`／`callback_status=p_status`。
- **【必須ゲート・v0.2で明記】`record_call_log`（anon実行口）はインターネットに直接晒さず、Twilio署名検証＋レート制限を行うCloud Run会話サーバ経由に限定する**。列長CHECK・priorityのclamp/CHECKは「壊れた/悪意ある入力1件あたりの被害」を抑える対策であり、大量リクエスト自体の防止（レート制限）はDB層の範囲外。指示書❹（音声接続）でこのanon口を公開する前に、会話サーバ側のTwilio署名検証＋レート制限の実装が必須条件。
- 【要確認/範囲外】録音音声の保存本体と自動削除（Twilio側・保持期間LOL未決）／折り返しリストの本格UI（営業所ホームへの組み込み）／受付テーブル（❶）との連携（`receipt_no`はFKなしのゆるい参照）。
- 実機確認（SQL Editor適用・件数突合・なりすまし5ロール＋anonの実測）は **LOL承認後に業務Aが実施**し、本メモに追記する。
